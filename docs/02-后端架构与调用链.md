# 后端架构与调用链

## 分层架构

```
HTTP 请求
  ↓
Handler 层  ← 解析请求参数（URL/Query/JSON），调用 Service，返回 JSON
  ↓
Service 层  ← 业务规则校验、事务控制、流程编排
  ↓
Repository 层 ← SQL 查询（参数化），只做数据访问
  ↓
PostgreSQL 数据库
```

## 各层职责边界

### Handler 层 (`backend/src/handler/`)

**只做三件事**：
1. 解析 HTTP 请求（Path 参数、Query 参数、JSON body）
2. 调用对应的 Service 方法
3. 将 Service 返回的结果包装成统一 JSON 格式 `{code, data, message}`

**绝对不写**：SQL、业务判断、事务控制。

示例文件：`product_handler.rs`, `inbound_handler.rs`, `inventory_handler.rs`

### Service 层 (`backend/src/service/`)

**负责**：
- 业务规则校验（如：SKU 是否重复、仓库是否存在、订单是否 draft）
- 事务控制（开始事务 → 调用多个 Repository → 提交/回滚）
- 流程编排（如：完成入库 = 写实收数量 + 增加库存 + 写流水 + 更新订单状态）

**绝对不写**：SQL 语句、HTTP 请求解析、JSON 序列化。

示例文件：`inbound_service.rs`, `inventory_service.rs`, `outbound_service.rs`

### Repository 层 (`backend/src/repository/`)

**只做一件事**：执行参数化 SQL 查询，返回 Rust 结构体。

**特点**：
- 所有 SQL 使用 `$1, $2, ...` 参数化，防止 SQL 注入
- 写操作支持事务上下文（接受 `Executor` trait）
- 不包含任何业务规则

示例文件：`inbound_repo.rs`, `inventory_repo.rs`, `product_repo.rs`

## 调用链示例：完成入库

```
POST /api/v1/inbound-orders/{id}/complete
  ↓
inbound_handler::complete()
  ↓ 解析 Path(id) + Json(CompleteInboundRequest)
  ↓
InboundService::complete()
  ↓ 1. 开启事务 (pool.begin())
  ↓ 2. 事务内读取订单 + 明细
  ↓ 3. 校验: status=draft, 全部明细已覆盖
  ↓ 4. 写入 actual_qty (update_item_actual)
  ↓ 5. 增加库存 (InventoryService::increase_stock_in_tx → InventoryRepository::upsert)
  ↓ 6. 写入库存流水 (TransactionRepository::insert)
  ↓ 7. CAS: draft → completed (cas_status)
  ↓ 8. 提交事务 (tx.commit())
  ↓
统一 JSON 响应 {code:0, data:{...}, message:"ok"}
```

## 事务与并发安全

### CAS (Compare-And-Swap)

项目使用 `cas_status` 方法保护订单状态流转：

```sql
UPDATE inbound_orders
SET status = 'completed'
WHERE id = $1 AND status = 'draft'
RETURNING *
```

如果返回空（None），说明订单已被其他请求改变了状态（如并发完成或取消），当前操作回滚。

### 库存操作的原子性

- **入库增加**：`INSERT ... ON CONFLICT DO UPDATE SET quantity = quantity + $delta` — 原子累加
- **出库扣减**：`UPDATE ... SET quantity = quantity - $delta WHERE quantity >= $delta` — 乐观锁，库存不足时返回 None

### quantity_before / quantity_after

库存流水表中的这两个字段记录了变更前后的库存数量：
- **增加时**：`before = after - delta`（从 upsert 返回的 after 反推）
- **减少时**：`before = after + delta`

这样做的好处是 `before` 和 `after` 始终与原子操作的结果一致，避免了读取-修改-写入之间的竞态窗口。

## 统一错误处理

`backend/src/error.rs` 定义了 `AppError` 枚举：

| 变体 | HTTP 状态码 | 业务码 | 示例场景 |
|------|-----------|--------|----------|
| Validation | 400 | 40001 | 参数校验失败 |
| NotFound | 404 | 40401 | 资源不存在 |
| Conflict | 409 | 40901 | SKU 重复 |
| BusinessRule | 422 | 42201 | 库存不足 |
| Internal | 500 | 50001 | 数据库错误 |

所有错误自动转换为统一 JSON：`{code, data:null, message:"..."}`。
