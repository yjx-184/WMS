# 出库流程说明

## 出库单生命周期

```
创建出库单 (draft)
  ↓ 可以编辑（修改仓库、明细）
  ↓ 表单会实时显示每行明细的可用库存
  ↓
完成出库 (completed)
  ↓ ① 填写每行实发数量
  ↓ ② 检查库存是否充足
  ↓ ③ 扣减库存
  ↓ ④ 写入库存流水 (change_type=outbound)
  ↓ ⑤ 订单状态 → completed
  ↓
取消出库 (cancelled)
  - draft 取消：直接改状态
  - completed 取消：回滚库存（增加回去）+ 写入回滚流水
```

## 与入库的关键区别

| 项目 | 入库 | 出库 |
|------|------|------|
| 库存操作 | **增加** | **扣减** |
| 库存不足 | 不存在（入库是增加） | 返回错误："库存不足: product=X location=Y need=Z avail=W" |
| 可用库存显示 | 不需要 | 表单中实时显示 |

## 1. 创建出库单

**端点**：`POST /api/v1/outbound-orders`

与入库流程相同：
1. 校验仓库/商品/库位存在且库位属于目标仓库
2. 生成单号（格式: `OUT + YYYYMMDD + 6位序列`）
3. 事务内插入主表和明细

前端表单会为每行明细**实时查询可用库存**，当计划数量超过可用库存时显示红色警告。

## 2. 完成出库

**端点**：`POST /api/v1/outbound-orders/{id}/complete`

**流程（全部在一个事务内）**：
1. 事务内读取订单 + 明细
2. **立即检查 status=draft**（尽早失败，避免多余操作）
3. 校验明细覆盖
4. 写入 `actual_qty`
5. **预检查库存**：逐行查询当前库存，若有不足 → 返回详细缺货列表，事务回滚
6. 逐行扣减库存（`InventoryRepository::decrease`，WHERE quantity >= delta）
7. 逐行写入库存流水（`change_type=outbound`）
8. CAS：`draft → completed`
9. 提交事务

## 3. 库存不足处理

出库完成时，如果在第 5 步预检查发现库存不足，返回的业务错误会包含**缺货明细**：

```
"库存不足: product=xxx location=yyy need=100 avail=50"
```

此时整个事务回滚，不会写入任何数据。

## 4. 取消出库单

### draft 取消
- CAS：`draft → cancelled`

### completed 取消
1. 事务内读取订单（确认仍为 completed）
2. 读取明细中的 `actual_qty`
3. 逐行**增加库存**（等于回滚之前的扣减）
4. 写入回滚流水（`change_type=inbound`，因为库存增加了）
5. CAS：`completed → cancelled`
6. 提交事务

注意：取消 completed 出库时，库存是**增加**回去的，所以流水类型是 `inbound`。

## 库存流水中的 quantity_before / quantity_after

出库完成时的每条流水记录：

```
before = after + actual_qty    （出库前库存是多少）
after  = decrease 返回的新库存  （出库后库存是多少）
```

例如：出库前库存为 100，实发 30 个 → `before=100, quantity=30, after=70`。
