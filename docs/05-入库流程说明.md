# 入库流程说明

## 入库单生命周期

```
创建入库单 (draft)
  ↓ 可以编辑（修改仓库、明细）
  ↓
完成入库 (completed)
  ↓ ① 填写每行实收数量
  ↓ ② 增加库存
  ↓ ③ 写入库存流水 (change_type=inbound)
  ↓ ④ 订单状态 → completed
  ↓
取消入库 (cancelled)
  - draft 取消：直接改状态
  - completed 取消：回滚库存 + 写入回滚流水
```

## 1. 创建入库单

**端点**：`POST /api/v1/inbound-orders`

**流程**：
1. 校验：仓库存在、商品存在、库位存在且属于目标仓库
2. 生成单号（格式: `IN + YYYYMMDD + 6位序列`）
3. 在事务内插入主表和明细
4. 返回完整入库单（含商品名称、SKU、库位编码等 JOIN 查询结果）

## 2. 编辑入库单

**端点**：`PUT /api/v1/inbound-orders/{id}`

**限制**：只能编辑 `draft` 状态的入库单

**流程**：
1. 事务内读取订单，校验 status=draft
2. 更新主表
3. 删除旧明细，插入新明细
4. CAS 验证：确认订单仍为 draft（防并发）
5. 提交事务

## 3. 完成入库

**端点**：`POST /api/v1/inbound-orders/{id}/complete`

**请求体**：每行明细的 `item_id` 和 `actual_qty`（实收数量）

**限制**：
- 只能是 `draft` 状态
- 必须填写所有明细
- actual_qty >= 0

**流程（全部在一个事务内）**：
1. 事务内读取订单 + 明细
2. 校验：全部明细已覆盖、item_id 合法、无重复
3. 逐行写入 `actual_qty`
4. 逐行增加库存（`InventoryRepository::upsert`）
5. 逐行写入库存流水（`change_type=inbound`）
6. CAS：`draft → completed`（`WHERE status='draft'`）
7. 提交事务

**并发保护**：第 6 步的 CAS 确保两个请求不会同时完成同一订单。

## 4. 取消入库单

**端点**：`POST /api/v1/inbound-orders/{id}/cancel`

### draft 取消
- CAS：`draft → cancelled`
- 不需要操作库存

### completed 取消
1. 事务内读取订单（确认仍为 completed）
2. 读取明细中的 `actual_qty`
3. 逐行扣减库存（等于回滚之前的增加）
4. 写入回滚流水（`change_type=outbound`）
5. CAS：`completed → cancelled`
6. 提交事务

## 库存流水中的 quantity_before / quantity_after

入库完成时的每条流水记录：

```
before = after - actual_qty    （入库前库存是多少）
after  = upsert 返回的新库存    （入库后库存是多少）
```

例如：入库前库存为 0，实收 100 个 → `before=0, quantity=100, after=100`。
如果再入库 50 个 → `before=100, quantity=50, after=150`。

这些值从原子 upsert 操作的返回值推导出来，保证与数据库实际状态一致。
